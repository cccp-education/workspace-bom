== TL;DR pour un LLM

Quand tu lis du code dans ce workspace et que tu vois `import education.cccp.contracts.context.*`,
tu sais que ces types viennent de `codebase-contracts:0.1.0` — un module N0 dans `workspace-bom`.

*NE PAS* créer de copies locales de ces types. La duplication a été éradiquée en Session 064.
La source unique de vérité est `foundry/public/workspace-bom/codebase-contracts/`.

== Pourquoi ce document existe

Session 063 a identifié un smell-pattern : les 3 types N0 `ContextChannel`, `ChannelBudget`,
`CompositeContext`, `CompositeContextConfig` étaient dupliqués dans 4 boroughs :
`codebase-gradle`, `planner-gradle`, `codex-gradle`, `training-gradle`.

Session 064 a résolu ce problème en créant un module Maven séparé, publié localement,
et en migrant tous les imports vers ce module. Ce document est la mémoire technique
de cette migration — pour qu'un LLM ou un développeur futur comprenne instantanément
l'architecture sans avoir à lire l'historique complet.

=== ContextChannel.kt

Sealed class avec 5 variants :

[cols="2,2,2,3"]
|===
| Variant | `source` | `budgetProportion` | `sectionHeader`

+ `ChannelBudget` data class (5 fractions de budget, doit sommer à 1.0)

+ `estimateTokens(text: String)` — heuristique tokens ≈ chars/3.5
+ `DEFAULT_TOKEN_BUDGET = 8000`

== Comment ajouter un nouveau borough consommateur

. Ajouter `mavenLocal()` dans les `repositories` du `build.gradle.kts`
. Ajouter `implementation("education.cccp:codebase-contracts:0.1.0")` dans les `dependencies`
. Importer : `import education.cccp.contracts.context.*`
. Utiliser les types directement — aucune copie locale nécessaire

== Comment modifier un type N0

. Modifier le fichier dans `workspace-bom/codebase-contracts/src/main/kotlin/...`
. Ajouter/mettre à jour les tests dans `codebase-contracts/src/test/kotlin/...`
. `./gradlew codebase-contracts:test` → tous les tests doivent passer
. `./gradlew publishToMavenLocal` → republie dans `~/.m2`
. Vérifier la compilation des boroughs consommateurs (planner, codex, training, codebase)
   → `./gradlew compileKotlin` dans chaque repo

== Reliquats connus (Session 064 → 065)

. *codex-gradle* : `org.jetbrains:annotations` version conflict (Gradle 9.5.1 strict)
  - Kotlin embedded `annotations:13.0` vs koog-agents `annotations:26.0.2-1`
  - Solution : `exclude("org.jetbrains", "annotations")` dans `implementation(libs.koog.agents)`
. *training-gradle* : `education.cccp.readme:0.0.1` plugin manquant
  - Solution : publier le plugin ou commenter la ligne dans `build.gradle.kts`
. *codebase-gradle* : `alias(libs.plugins.kotlin.jvm)` non résolu dans le bloc `plugins {}`
  - Solution : remplacer par `id("org.jetbrains.kotlin.jvm") version "2.3.20"`
. *planner-gradle* : compile OK depuis Session 064 — seul borough N2 pleinement fonctionnel

Ces bugs sont *préexistants* à la migration N0. La restructuration n'a rien cassé.
La Session 065 est programmée pour les résoudre (cf. `PROMPT_REPRISE.adoc`).

== Règles pour les LLM futurs

. *Quand un LLM propose d'ajouter un type contextuel* → le mettre dans `codebase-contracts`, PAS dans un borough individuel
. *Quand un LLM voit `cccp.vibecoding.contracts.context`* → c'est du code legacy/obsolete, migrer vers `education.cccp.contracts.context`
. *Quand un borough ne compile pas* → vérifier d'abord `~/.m2/repository/education/cccp/codebase-contracts/0.1.0/`, lancer `./gradlew publishToMavenLocal` dans workspace-bom si absent
. *Quand la version change* → mettre à jour les `implementation("education.cccp:codebase-contracts:X.Y.Z")` dans les 4 boroughs consommateurs
. *Ne JAMAIS recréer `cccp/vibecoding/contracts/context/`* — ce répertoire a été supprimé dans les 4 boroughs et ne doit pas revenir

== Bootstrap pour un nouveau développeur

```bash
# 1. Publier les contrats en local
cd foundry/public/workspace-bom
./gradlew publishToMavenLocal

# 2. Vérifier que les contrats sont disponibles
ls ~/.m2/repository/education/cccp/codebase-contracts/0.1.0/

# 3. Lancer les tests des contrats
./gradlew codebase-contracts:test       # 16/16 PASS
./gradlew agent-contracts:cucumberTest  # 36/36 PASS

# 4. Compiler un borough consommateur (exemple planner)
cd ../planner-gradle
./gradlew :planner-plugin:compileKotlin # doit compiler sans erreur
```
